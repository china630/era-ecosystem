## C. Финансы и ERP

| # | Вопрос | Ответ |
|---|--------|-------|
| C1 | Какой **ERP** (1C, SAP, другое)? Есть **API**? | У **Nafta сейчас 1C**. Для **ERA Hotel PMS** целевое ядро — **ERA Finance** (отдельный репозиторий). API OpenAPI — после заморозки scope |
| C2 | Кто владелец **кодов дохода** — PMS или ERP? | **Гибрид (рекомендация зафиксирована):** операционные коды (`ROOM`, `FOOD`, `MEDICAL`) — **мастер в сателлите**; маппинг на GL/NAS — **в ERA Finance** (или стороннем ERP). | C3 | Нужен ли обратный статус «**проведено в GL**» в PMS? | **Да, read-only** (статус e-qaimə / `glPosted`) — опционально в ф.1, без проводок в PMS |
| C4 | Достаточно ли пакета **E1 раз в сутки** после night audit? | **Real-time + E1:** folio charge/payment/void/check-out → сразу в ERP; **E1** после night audit — сверка дня. См. [17-az-compliance.md](17-az-compliance.md) |

## H. Архитектура: PMS, era-fb-pos, ERP (зафиксировано 2026-05)

### Термины

| Термин | Расшифровка | У нас |
|--------|-------------|--------|
| **POS** | Point of Sale — касса точки продаж | Ресторан/бар, SPA-касса, quick posting; **не** модуль розничного магазина |
| **era-fb-pos** | **F&B** (Food & Beverage) POS — сателлит ресторана | Карта зала, чеки, KDS, Z-отчёт; ресторан **в отеле и вне** |
| **STOCK** | Склад Elektraweb | **Только ERA Finance / ERP**, не операционный склад в PMS |

### Два слоя (не «или PMS, или ERP»)

```
Слой A — операции отеля (era-hotel-pms)
  бронь, in-house, folio, баланс, check-out, night audit

Слой B — учёт (ERA Finance / 1C Nafta)
  GL, склад, закупки, e-qaimə submission, закрытие периода
```

См. [01-finance-boundary.md](01-finance-boundary.md).

### Когда нужна интеграция **hotel-pms** (не только финядро)

| Сценарий | hotel-pms | ERA Finance |
|----------|-----------|-------------|
| Ресторан в отеле, оплата **«на номер»** | **Да** — `room-charge` → folio | Да — E1/E2, фискал |
| Ресторан в отеле, **наличные на месте** | Опционально (folio payment) | Да — KKM + выручка |
| Ресторан **вне отеля** (walk-in) | **Нет** | Да |
| Мелочь SPA/напитки с **ресепшена** | **Да** — quick posting (WA0135) | Да |
| **Мед. процедуры** (санаторий) | **Да** — медконтур + folio при доплатах; пакет в тарифе — без лишних charge | Да |
| Склад, рецептуры, PO | **Нет** | Да |

Мост в коде: [23-pos-bridge.md](23-pos-bridge.md) (`POST /api/pos/room-charge`, календарь слотов).

### SPA внутри отеля — quick posting

Пока **нет** отдельной SPA-кассы (A1):

- **Оставляем quick posting** в `era-hotel-pms` — продажи SPA-товаров/услуг на folio in-house без зала.
- **Medical UI** — процедуры и alerts; доплаты — MED-04 на folio.
- **POS calendar** в PMS — только **бронь кабинета/слота**, не полная касса.

### Нужна ли отдельная сателлита **era-spa-pos**?

**Для Nafta-санатория на текущем этапе — нет.** Причины:

1. Мед. контур уже в **era-hotel-pms**; услуги по гипотезе D3 **включены в номер**, не дублируем второй продукт.
2. A1: полноценная SPA-касса Elektraweb **не используется**.
3. Отдельный сателлит оправдан, если появится **отдельный бизнес**: SPA-ритейл с кассой, несколько outlets, KKM на стойке SPA, сложное меню как в ресторане — тогда либо **outlet в era-fb-pos**, либо редкий **era-spa-pos** (только при явном scope).

**Рекомендация:**

| Приоритет | Решение |
|-----------|---------|
| Сейчас | PMS: мед + quick posting + календарь слотов |
| Ф.2 ресторан | **era-fb-pos** + мост room-charge |
| Позже | SPA-касса → сначала оценить **outlet SPA в era-fb-pos**; отдельный `era-spa-pos` — только если F&B-модель не подходит |
