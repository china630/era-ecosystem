Руководство пользователя ERA Finance для бухгалтера

Версия документа соответствует продуктовым требованиям (PRD.md, §4) и техническому заданию (TZ.md). Ниже описан фактический интерфейс веб-приложения apps/web и то, как система считает и проводит ключевые операции.

Иллюстрации в каталоге images/ — эталонные макеты интерфейса. Их можно заменить скриншотами вашего стенда (см. README.md).

1. Вход, организация и роли

1.1. Вход в систему

Откройте в браузере адрес веб-приложения (локально: http://localhost:3000). На странице «Вход» укажите email и пароль. При успешной авторизации вы попадаете в рабочую оболочку с боковым меню и шапкой.

1.2. Организация (тенант)

В системе одна активная организация = одно юрлицо (VÖEN). Если у вас несколько компаний:

  • В шапке доступен переключатель организаций (если у пользователя несколько доступов).
  • Раздел «Мои компании» (/companies) — управление доступом и создание новой организации (в пределах квоты тарифа).

Все документы, проводки и отчёты строятся только в контексте выбранной организации.

1.3. Роли, важные для бухгалтера

Роль Типичные задачи
Owner Подписка, полный доступ, закрытие периода (совместно с Admin).
Admin Пользователи, настройки, закрытие периода.
Accountant Финансы, счета, банк, отчёты, зарплата, проводки — основной профиль мануала.
User Ограниченный доступ; часть мутаций (например, оплаченные счета, ручной журнал) может быть запрещена политиками RBAC.

Закрытие учётного периода доступно только Owner и Admin (см. §9).

2. Главная страница и переключатели

2.1. Что показывает дашборд

После входа открывается «Главная» (/). Для авторизованного пользователя подгружаются виджеты (данные с API дашборда):

  • Остаток денег (агрегат по счетам кассы/банка в выбранной книге учёта).
  • Баланс по налоговым/спецсчетам (ориентир для контроля, в т.ч. счёт 521 в логике виджета).
  • Выручка по дням (график за последние дни).
  • Топ товаров по количеству в движениях.
  • Топ дебиторов и кредиторов по остаткам взаиморасчётов.

Задачи бухгалтера: быстрая оценка ликвидности, дебиторки/кредиторки и динамики выручки без построения полного отчёта.

2.2. Переключатель NAS / IFRS

В шапке (слева) — тумблер NAS и IFRS. Он задаёт активную книгу (ledgerType) для запросов отчётности и дашборда: план счетов и проводки могут вестись параллельно, если включена поддержка Multi-GAAP в вашей конфигурации.

2.3. Быстрые действия (+)

Кнопка «+» открывает меню:

  • Новый счёт (/invoices/new)
  • Быстрый расход (/expenses/quick)
  • Новый сотрудник (/employees/new)
  • Выпуск из производства (если модуль не заблокирован тарифом)

На узких экранах то же меню доступно через плавающую кнопку внизу справа.

2.4. Демо и подписка

При пробном тарифе может отображаться баннер срока действия демо (логика v4 в PRD). После истечения без оплаты интерфейс может перейти в режим только просмотр (мутации блокируются API).

3. Продажи: счета, контрагенты, номенклатура

3.1. Контрагенты (/counterparties)

Задача: единый справочник клиентов и поставщиков.

Фишки:

  • Обязательные атрибуты включают VÖEN (10 цифр), наименование, адрес, банковские реквизиты (по ТЗ).
  • Тип взаимодействия: клиент / поставщик; в учёте используются счета 211 (дебиторка) и 531 (кредиторка).

Как работает учётно: при проводках с привязкой к контрагенту система обновляет обороты и сальдо по соответствующим счетам плана счетов организации.

3.2. Товары и услуги (/products)

Задача: каталог для строк счетов: SKU, цена, ставка НДС.

НДС в MVP: только 0%, 18% и освобождён (спецрежимы — по дорожной карте).

3.3. Счета (/invoices, /invoices/new, /invoices/[id])

Жизненный цикл статусов: Draft → Sent → Paid / Overdue / Cancelled.

Ключевая бухгалтерская логика (при переходе в Paid):

  1. Создаётся транзакция (группировка проводок).
  2. Формируются проводки: Дт 101/221 (касса или банк) — Кт 211 (погашение дебиторской задолженности по выручке).

То есть оплата счёта не дублирует выручку, а закрывает дебиторку, если выручка была начислена ранее (логика отображения и статусов на экране счёта согласована с API).

Фишки интерфейса списка счетов:

  • Перевод в Sent / Paid с панели действий.
  • Частичная оплата: ввод суммы и даты → POST .../payments — создаётся оплата с распределением на инвойс; статусы PARTIALLY_PAID / PAID зависят от суммы оплат относительно суммы счёта.
  • Отправка на email (если настроен SMTP на стороне API).

PDF: печатная форма счёта доступна на AZ / RU / EN (генерация на стороне API).

4. Банк и выписки

Раздел «Банк» (/banking).

4.1. Задачи модуля

  • Загрузка выписок CSV / Excel (MVP).
  • Просмотр строк выписки, сопоставление (match) с инвойсами и создание операций расхода.
  • При наличии модуля banking direct в подписке — синхронизация с банком (кнопка синхронизации, статус последнего запуска, webhook URL в статусе).

4.2. Как это стыкуется с учётом

Строка выписки после сопоставления помечается как matched; связь с инвойсом позволяет не проводить оплату дважды. Несопоставленные строки остаются в работе — фильтр «Только требующие внимания» сужает список.

4.3. Мультивалюта

Ведение счетов в AZN, USD, EUR; курсы ЦБА (cbar.az) подгружаются по регламенту (ежедневно). Переоценка валютных остатков — регламентная операция на конец месяца (не ежедневный пересчёт по умолчанию).

Часть возможностей прямого банка может быть закрыта замком в меню, если тариф не включает bankingPro.

5. Быстрый расход

Раздел /expenses/quick.

Задача: оперативно отразить типовой расход (аренда, налоги и т.п.) с формированием проводок через API учёта.

Как работает: вы заполняете поля формы (счета, сумма, дата, назначение); сервис создаёт транзакцию и проводки с проверкой баланса Дт = Кт. Удобно для мелких повторяющихся платежей без полного платёжного поручения.

6. Персонал: сотрудники, структура, зарплата

6.1. Сотрудники (/employees)

Кадровые карточки: ФИО, ФИН, договор, связь с должностью и подразделением (при включённой оргструктуре). Создание карточки может проверять штатные лимиты (quota / staffing).

6.2. Структура (/hr/structure)

Дерево подразделений, штатные позиции (количество ставок, вилка окладов), назначение руководителя узла. Нужно для аналитики по ЦФО в P&L (разрез по департаменту), если проводки по зарплате связаны с сотрудником и позицией.

6.3. Зарплата (/payroll)

Задачи: расчётный период (год/месяц), запуск расчёта (часто асинхронно через очередь при большом числе сотрудников), просмотр расчётных листков, вкладка отсутствий (отпуск / больничный в календаре).

Как считает (локализация АР, упрощённо по ТЗ):

Компонент Ориентир
Подоходный налог 14% с учётом необлагаемого минимума (8000 AZN в нефтяном секторе — уточняйте актуальные ставки в нормативке).
DSMF 3% с работника, 22% (или альтернативная схема 10% + фикс) с работодателя.
ITS (медстрах) 2% + 2%.
Безработица 0,5%.

Проводки при начислении: типовая схема Дт 721 (расходы на оплату труда) — Кт 533 (задолженность перед персоналом), в рамках транзакции БД.

Роль User может не видеть разрушающих действий (кнопки удаления и т.д.) — зависит от политики.

7. Основные средства

Раздел /fixed-assets: учёт объектов ОС, ввод остатков, амортизация.

Фишка закрытия месяца: при закрытии периода с главной или отчётов система может начислить амортизацию по объектам и сформировать проводки (ответ API сообщает, были ли затронуты объекты и на какую сумму).

Если пункт меню с замком — модуль не входит в ваш тариф (fixedAssets).

8. Склад и производство

8.1. Склад (/inventory)

Задачи: склады, остатки, закупка, перемещение, корректировки, инвентаризация (аудиты).

Корректировка остатков (adjustStock): изменение количества в одной БД-транзакции с проводками по отклонению себестоимости (счета ТМЦ / себестоимость / прочие доходы-расходы — по политике, см. PRD §4.10).

8.2. Производство (/manufacturing)

Рецептуры и выпуск. Может быть недоступно по подписке (замок в меню и быстрых действиях).

9. Отчётность: ОСВ, P&L, закрытие периода

Страница /reporting.

9.1. Оборотно-сальдовая ведомость (ОСВ)

  • Задаёте период (dateFrom / dateTo).
  • Кнопка загрузки ОСВ запрашивает /api/reporting/trial-balance с учётом текущей книги NAS/IFRS.
  • Таблица: счёт, дебет/кредит входящие, обороты, исходящие сальдо.

Задача бухгалтера: сверка счетов, подготовка к закрытию месяца, поиск перекосов.

9.2. Отчёт о прибылях и убытках (P&L)

  • Методология: начисление (accrual) — выручка по отгрузке, как описано в подсказках API.
  • Опционально: фильтр подразделения для разреза по ЦФО (список департаментов подгружается с API).

Итоговая строка — чистая прибыль за период.

9.3. Закрытие периода

Блок закрытия месяца (год/месяц) доступен Owner/Admin. Запрос POST /api/reporting/close-period:

  • Фиксирует закрытый период (новые проводки за закрытый месяц запрещены).
  • Может запустить амортизацию ОС за период (см. ответ сервера).

10. Дебиторка, старение, акт сверки и взаимозачёт

Раздел Путь Задача
Дебиторка /reporting/receivables Список задолженностей покупателей, остатки по 211.
Старение /reporting/aging Интервалы 0–30 / 31–60 / 61–90+ дней по просрочке.
Акт сверки /reporting/reconciliation Сверка оборотов 211/531 по контрагенту; блок встречных требований для взаимозачёта.

Взаимозачёт

Если по контрагенту одновременно есть дебиторка (211) и кредиторка (531), система предлагает сумму зачёта = min(остатки). При подтверждении создаётся одна транзакция с проводкой Дт 531 — Кт 211 и распределением по инвойсам через строки оплат (без дублирования второй пары проводок). На карточке инвойса может быть действие «Оплатить зачётом».

11. Экспорт для налоговой (шаблон)

Страница /reporting/tax-export: формирование файлов под сдачу / приложения к отчётности (движок экспорта, шаблоны Excel и т.д. — по мере развития модуля).

Прямая отправка в e-taxes.gov.az относится к дорожной карте v3 (интеграция), а не к минимальному MVP.

12. Настройки: команда, аудит, подписка, маппинг NAS↔IFRS

Раздел Кто видит Назначение
Команда и доступ /settings/team Owner, Admin Пользователи организации, роли, приглашения.
Аудит /settings/audit Owner, Admin Журнал мутаций; проверка целостности хешей (integrity-check).
Подписка /settings/subscription Все авторизованные (просмотр); оплата по политике Тариф, модули, квоты, демо.
Маппинг NAS ↔ IFRS /accounting/mapping По тарифу (ifrsMapping) Соответствие счетов между книгами.

Неизменяемый аудит (хеш SHA-256, архив старше года) описан в PRD §4.8 и TZ §9.

13. Частые вопросы и ограничения

Почему пункт меню с замком? Модуль не входит в activeModules подписки (feature gating).

Почему не сохраняется операция? Возможны: истёкшая подписка / READ_ONLY, закрытый период, превышение квоты, ошибка валидации DTO (строгий режим API).

Где валюта и время? В БД — UTC, отображение и AZN — локаль Азербайджана (PRD §12).

Кто может менять проводки? В MVP правки журнала допускаются с записью в аудит; политика «только сторно» — на будущее.

Конец руководства. Источники: PRD.md, TZ.md, код apps/web и apps/api.
